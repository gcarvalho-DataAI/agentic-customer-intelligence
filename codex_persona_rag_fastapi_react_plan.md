# Codex Implementation Plan — Persona RAG Layer with FastAPI + React

## Project

Repository name:

```text
agentic-customer-intelligence
```

Project description:

```text
End-to-end AI system combining customer clustering, synthetic personas, persona-specific RAG, FastAPI backend, and React frontend for conversational business intelligence.
```

---

# Objective

Implement the agentic persona layer using:

```text
FastAPI backend
React frontend
PostgreSQL + pgvector vector store
persona-specific RAG
cluster_profiles.json generated by the ML module
OpenAI-compatible LLM client
```

The goal is to build a demonstrable application where stakeholders can:

1. View the generated customer personas.
2. Select one or more personas.
3. Ask business questions.
4. Receive side-by-side responses from each selected persona.
5. See which retrieved context was used by each persona.
6. Understand that each answer is grounded in the cluster profile plus persona-specific RAG documents.

---

# Important Design Principle

The RAG layer must not replace the statistical cluster profile.

The cluster profile is the primary source of truth.

RAG documents are used only to enrich:

```text
tone of voice
behavioral interpretation
price sensitivity reasoning
promotion response patterns
channel-switching reasoning
consumer behavior context
business-oriented recommendations
```

The agent must never invent new statistics, percentages, or demographic claims from RAG documents.

# Why RAG Is Useful Here

The cluster profiles provide statistical structure, but they do not contain all contextual knowledge needed to answer business questions.

For example, if a user asks:

```text
Como vocês reagiriam a um aumento de preço de 10%?
```

The agent should combine:

```text
cluster profile
+
differentiators versus base
+
retrieved context about price sensitivity, loss aversion, cashback, loyalty, channel switching, and consumer habit
+
guardrails
```

The answer should be practical, business-oriented, and explicitly consistent with the persona profile.

# High-Level Architecture

```text
React UI
  ↓
FastAPI backend
  ↓
Persona service
  ↓
Retriever service
  ↓
PostgreSQL + pgvector vector store
  ↓
Prompt builder
  ↓
LLM client
  ↓
Persona answer response
```

Detailed flow:

```text
User selects personas in React
↓
User sends business question
↓
React calls POST /api/chat
↓
FastAPI loads selected persona profiles
↓
For each persona:
    retrieve persona-specific + shared context
    build grounded prompt
    call LLM
    return answer + retrieved context
↓
React renders persona responses side by side
```

Operational ports:

```text
Local OpenAI-compatible LLM API: http://127.0.0.1:8000/v1
Persona RAG FastAPI backend: http://127.0.0.1:8010/api
React frontend: http://localhost:5173
```

```python
class PersonaProfile(BaseModel):
```

```python

```

Rules:

For selected persona `0`, retrieve from:

```text
persona_id = persona_0
OR
persona_id = shared
```

Simple implementation:

```text
retrieve top 2 persona-specific chunks

```

Do not retrieve documents from other personas.

If no persona-specific documents are found, fallback to shared only.

Suggested prompt:

```text
You are a synthetic persona agent derived from a customer cluster.

You are not a real individual customer.
You represent a statistical segment.

Primary source of truth:
Use the statistical cluster profile as the main source of truth.

Secondary context:
Use retrieved context only to enrich reasoning, tone, and business interpretation.

Do not:
- invent new statistics
- make sensitive or stereotypical claims
- claim that external research proves this exact cluster behaves a certain way
- ignore uncertainty when the evidence is weak

Answer style:
- answer in first person plural
- be practical and business-oriented
- explain trade-offs clearly
- mention uncertainty when the profile does not support a strong conclusion
- keep the response concise but useful

Persona:
{segment_name}

Cluster profile:
{profile}

Differentiators versus overall base:
{differentiators}

Business interpretation:
{business_interpretation}

Retrieved context:
{retrieved_context}
```

```markdown
# Persona RAG Architecture

## Objective

The RAG layer enriches synthetic persona agents with contextual knowledge while preserving cluster profiles as the source of truth.

## Why RAG

Cluster profiles describe the segment statistically. RAG adds contextual knowledge such as behavioral science principles, pricing sensitivity, loyalty programs, and channel-switching patterns.

## Architecture

React frontend → FastAPI backend → persona loader → retriever → PostgreSQL + pgvector → prompt builder → LLM client.

## Retrieval Strategy

Each persona retrieves from:
- its own persona-specific documents
- shared documents

Documents from other personas are not retrieved.

## Guardrails

- Do not invent statistics.
- Do not make stereotypical claims.
- Do not overclaim external research.
- Mention uncertainty when evidence is weak.
- Treat behavioral science as qualitative support, not proof.

## Limitations

- Synthetic data.
- RAG documents are curated and not exhaustive.
- Behavioral science principles are general.
- Persona responses should be validated with real customer research before production use.

## Next Steps

- Add real customer interviews.
- Add market reports.
- Add brand guidelines.
- Add evaluation framework for persona consistency.
- Track retrieval quality and answer grounding.
```

# Code Quality Requirements

Use:

```text
small functions
clear names
typed schemas
simple error handling
relative paths
environment variables
clean separation between backend and frontend
mock response when no LLM key is configured
```

Avoid:

```text
hardcoded absolute paths
retrieving documents from wrong persona
putting all context directly in frontend
overengineering graph orchestration
claiming scientific certainty
inventing statistics
```

# Important Presentation Message

The implementation should support this explanation:

```text
The ML layer creates statistical personas from clustering. The RAG layer does not replace those profiles; it enriches each persona with contextual knowledge about consumer behavior, pricing, loyalty, and channel preferences. Each answer is generated from the cluster profile plus retrieved persona-specific and shared context, with guardrails to prevent unsupported claims.
```

This is the main architectural story of the agentic layer.
